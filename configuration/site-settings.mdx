KYA Hukuk stores dynamic site configuration in the `SiteSetting` database model. You can manage these settings through the admin panel at `/admin/ayarlar`.

## SiteSetting model

The `SiteSetting` model stores key-value pairs for site-wide configuration.

### Schema

```python models.py
class SiteSetting(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    key = db.Column(db.String(100), unique=True, nullable=False)
    value = db.Column(db.Text, nullable=True)
    label = db.Column(db.String(200), nullable=True)  # Admin panel label
    type = db.Column(db.String(30), default='text')   # text, textarea, image, email, phone
```

## Caching

Settings are cached in memory for 60 seconds to reduce database queries.

### Cache implementation

```python app.py
_settings_cache: dict = {}
_settings_cache_ts: float = 0.0
_SETTINGS_TTL = 60  # seconds

def site_settings() -> dict:
    global _settings_cache, _settings_cache_ts
    if time.time() - _settings_cache_ts < _SETTINGS_TTL and _settings_cache:
        return _settings_cache
    
    _settings_cache = {s.key: s.value for s in SiteSetting.query.all()}
    _settings_cache_ts = time.time()
    return _settings_cache
```

### Cache invalidation

The cache is cleared when settings are updated:

```python app.py
def _invalidate_settings_cache():
    global _settings_cache_ts
    _settings_cache_ts = 0.0

# Called after saving settings:
SiteSetting.set('logo_url', new_url)
_invalidate_settings_cache()
```

<Note>
  Without manual invalidation, changes take up to 60 seconds to appear. The admin panel automatically invalidates the cache after updates.
</Note>

## Template access

All settings are available in templates via the `settings` context variable:

```python app.py
@app.context_processor
def inject_settings():
    return {'settings': site_settings()}
```

**Usage in templates:**

```jinja
{{ settings.get('contact_phone', 'No phone') }}
{{ settings.get('contact_email') }}
{{ settings.get('footer_text') }}
```

- `text` - Single-line text input
- `textarea` - Multi-line text input
- `email` - Email input with validation
- `image` - File upload with GitHub storage

## Admin panel

Settings are managed at `/admin/ayarlar`.

    - **Logo & Marka** - Logo uploads and URLs

  <Step title="Save changes">

  </Step>
</Steps>

### Logo management

Two special endpoints for logo management:

**Upload logos:**
```python
POST /admin/ayarlar
Content-Type: multipart/form-data

logo_file: <file>
logo_white_file: <file>
```

Uploaded files are stored via GitHub and URLs are updated automatically.

**Reset to defaults:**
```python
POST /admin/logo-sifirla
```

Resets `logo_url` and `logo_white_url` to `/Assets/logo-color.webp` and `/Assets/logo-disi.webp`.

## Migration and defaults

Default settings are inserted during `init_db()` (app.py:769):

```python
defaults = {
    'contact_address': 'Balgat Mahallesi...',
    'contact_phone': '+90 312 123 45 67',
    # ...
}

existing_settings = {s.key: s for s in SiteSetting.query.all()}
for key, value in defaults.items():
    if key not in existing_settings:
        db.session.add(SiteSetting(key=key, value=value))
```

**Behavior:**
- Only inserts missing keys (doesn't overwrite existing values)
- Safe to run multiple times
- Add new settings by updating the `defaults` dict

### WebP logo migration

Old `.png` logos are automatically migrated to `.webp`:

```python app.py
_MIGRATIONS = {
    'logo_url':       ('/Assets/logo-color.png',  '/Assets/logo-color.webp'),
    'logo_white_url': ('/Assets/logo-disi.png',   '/Assets/logo-disi.webp'),
}

for key, (old_val, new_val) in _MIGRATIONS.items():
    row = existing_settings.get(key)
    if row and row.value == old_val:
        row.value = new_val
```

This runs during every `init_db()` call.
